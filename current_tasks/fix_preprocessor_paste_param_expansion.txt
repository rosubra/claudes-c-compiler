Fix preprocessor: expand macro args used both with and without ## operator

Bug: When a macro parameter appears both with ## (token paste) and without ##
in the replacement body, the current code skips argument expansion for ALL
occurrences. Per C11 ยง6.10.3.1, only occurrences adjacent to ## should use
the raw (unexpanded) argument; other occurrences should use the expanded form.

Example:
  #define __bitf(src, bitfield, x) (x >> src << (0 ? bitfield))
  #define bitfs(bitfield, s) __bitf(bitfield, bitfield##_##s)
  #define POWER_LEVEL 0, 2:0
  #define POWER_LEVEL_NORMAL 0x3
  bitfs(POWER_LEVEL, NORMAL)

Expected: __bitf(0, 2:0, 0x3) -> (0x3 >> 0 << (0 ? 2:0))
Actual:   __bitf(POWER_LEVEL, POWER_LEVEL_NORMAL) -> wrong expansion

Fix: Always expand all arguments in prescan step. handle_stringify_and_paste
already receives raw args for ## operations, so removing the paste_params
check from the prescan step is the correct fix.

This blocks RISC-V defconfig kernel build (drivers/gpu/drm/i2c/ch7006_drv.c).
