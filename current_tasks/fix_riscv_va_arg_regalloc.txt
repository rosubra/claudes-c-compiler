Fix RISC-V va_start/va_arg/va_copy to handle register-allocated values

Problem: emit_va_start, emit_va_arg, and emit_va_copy in the RISC-V backend
directly access stack slots via get_slot/is_alloca without checking if the
value (va_list_ptr) has been register-allocated to a callee-saved register.

When the IR lowering produces a GEP result for the va_list address, and the
register allocator assigns that value to a callee-saved register (e.g., s1),
the va_start/va_arg/va_copy code loads from the (uninitialized) stack slot
instead of reading from the register. This causes a segfault.

Fix: Add a helper method to resolve va_list_ptr values to a target register
by checking register allocation first, falling back to slot access.

This will fix 100% failure rate on va-arg and va-list tags in RISC-V tests.
