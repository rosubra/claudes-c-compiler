Task: Fix all hardcoded LP64 type sizes throughout frontend/IR for i686 (ILP32) support
Status: in_progress

Problem:
Multiple layers of the compiler hardcode LP64 (64-bit) sizes for sizeof, alignof, and type
mapping, causing sizeof(long)==8, sizeof(void*)==8, and other incorrect results on i686.

The core CType system (common/types.rs) correctly uses target_ptr_size(), but the parser,
sema, and IR lowering layers bypass CType and use hardcoded 8 values.

Files to fix:
- src/frontend/parser/parser.rs: sizeof_type_spec() and alignof_type_spec()
- src/frontend/sema/const_eval.rs: sizeof_type_spec()
- src/ir/lowering/types.rs: type_spec_to_ir(), scalar_type_size_align(), compute_decl_info()
- src/ir/lowering/expr_types.rs: sizeof result types and hardcoded sizes
- src/ir/lowering/global_init_compound*.rs: hardcoded pointer sizes
- src/backend/call_emit.rs: hardcoded 8 for stack slot sizes
- src/ir/lowering/types_seed.rs: int64_t mapped to Long (wrong on ILP32)
